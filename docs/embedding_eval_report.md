# 임베딩 성능 평가: HuggingFace vs OpenAI

Eden의 임베딩 백엔드 후보인 **HuggingFace(`jhgan/ko-sroberta-multitask`)**와 **OpenAI(`text-embedding-3-small`)**를, 같은 성경 구절 코퍼스·같은 테스트 질문으로 검색 정확도를 직접 비교한 평가입니다.

## 요약

| 항목 | 내용 |
|---|---|
| 코퍼스 풀 | 16,500개 구절 |
| 테스트 질문 | 40개 (GPT-4o-mini로 생성) |
| HuggingFace 모델 | `jhgan/ko-sroberta-multitask` |
| OpenAI 모델 | `text-embedding-3-small` |
| 평가 지표 | Recall@1 / Recall@3 / Recall@5 / MRR |
| Seed | 42 |

| 지표 | HuggingFace | OpenAI | 차이 (HF − OA) |
|---|---:|---:|---:|
| Recall@1 | **0.425** | 0.375 | +0.050 |
| Recall@3 | **0.675** | 0.625 | +0.050 |
| Recall@5 | **0.750** | 0.700 | +0.050 |
| MRR | **0.5579** | 0.4912 | +0.0667 |

이 테스트에서는 한국어 특화 모델인 HuggingFace `ko-sroberta-multitask`가 모든 지표에서 OpenAI `text-embedding-3-small`보다 근소하게 앞섰습니다.

---

## 1. 평가 방법론

1. 기존에 로컬로 구축돼 있던 HuggingFace(`ko-sroberta-multitask`) Chroma 컬렉션의 성경 구절 **16,500개**를 검색 후보 코퍼스 풀로 사용했습니다.
2. 이 중 족보·율법 목록 위주 책(레위기·민수기·역대상·역대하·에스라·느헤미야)은 "질문 생성 대상"에서 제외했습니다 — 자연어 고민 질문과 의미적으로 매칭되기 어려운 내용이라 테스트 질문의 근거로 쓰기 부적합하기 때문입니다. 단, 검색 후보 풀에는 그대로 남겨 실제 서비스와 동일하게 노이즈로 작용하게 했습니다.
3. 제외되지 않은 구절 중 **40개**를 무작위(seed=42)로 뽑아, GPT-4o-mini에게 "이 구절을 몰라도 위로/답이 될 만한 고민을 가진 사람이 챗봇에게 물어볼 법한 한국어 질문"을 하나씩 생성하도록 했습니다. 구절 문장을 그대로 인용하지 않되, 검색으로 원 구절을 다시 찾아낼 수 있을 만큼 핵심 상황·인물·키워드는 남기도록 지시했습니다.
4. 같은 16,500개 구절 원문을 HuggingFace와 OpenAI 두 모델로 각각 새로 임베딩했습니다.
5. 각 테스트 질문을 두 모델로 임베딩한 뒤, 코사인 유사도로 코퍼스 풀 전체에서 상위 5개를 검색해 정답 구절이 몇 위 안에 들어오는지 확인했습니다.
6. **Recall@1/3/5** (정답이 top-k 안에 들어온 질문의 비율)와 **MRR**(Mean Reciprocal Rank, 정답 순위의 역수 평균)을 산출했습니다.

> ⚠️ 로컬 Chroma DB 자체의 HNSW 벡터 인덱스는 패키지 버전 불일치로 손상되어 있어(`Error loading hnsw index`) Chroma의 유사도 검색 API는 사용하지 않았습니다. 대신 SQLite(`chroma.sqlite3`)에 저장된 원문 텍스트만 직접 추출해, 두 임베딩 모두 동일한 조건(같은 텍스트·같은 검색 로직)에서 새로 계산하는 방식으로 공정성을 확보했습니다.

## 2. 결과 해석

HuggingFace `ko-sroberta-multitask`가 Recall@1/3/5에서 각각 **+5.0%p**, MRR에서 **+0.067**만큼 OpenAI `text-embedding-3-small`보다 높게 나타났습니다. `ko-sroberta-multitask`는 한국어 문장 간 의미 유사도(STS) 과제로 파인튜닝된 모델이라, 한국어 질문–구절 매칭에서 범용 다국어 모델인 OpenAI 임베딩보다 근소하게 유리했던 것으로 보입니다.

40개 질문 중 정답이 top-5 안에 들어온 비율은 HuggingFace 75.0%, OpenAI 70.0%로 두 모델 모두 실용적인 수준의 검색 정확도를 보였습니다.

## 3. 한계 및 유의사항

이번 평가에 쓰인 로컬 코퍼스는 66권 중 **20권만 담긴 부분 빌드**입니다(구약 역사서·율법서 위주, 신약 미포함). 반면 실제 프로덕션(Qdrant Cloud, OpenAI 임베딩)은 전체 66권 **31,077개** 구절을 다 담고 있습니다.

즉 이 점수는 **"같은 텍스트를 줬을 때 어느 임베딩이 더 정확히 찾아내는가"**를 보는 순수 검색 정확도 비교이며, 커버리지·클라우드 배포 편의성까지 포함한 "현재 서비스 중인 백엔드의 우열"을 뜻하지는 않습니다.

## 4. 결론

동일 코퍼스 기준 순수 임베딩 정확도는 **HuggingFace(`ko-sroberta-multitask`)가 근소하게 우세**했습니다(Recall@1 기준 42.5% vs 37.5%). 다만 Eden은 로컬 자원 없이도 클라우드에서 안정적으로 서비스하기 위해 전체 성경(31,077구절)을 사전 임베딩해 Qdrant Cloud에 올려둔 OpenAI 백엔드를 프로덕션에 사용 중이며, 이번 결과는 향후 한국어 특화 임베딩으로 전환할 경우의 잠재적 정확도 개선 여지를 보여주는 참고 지표로 활용할 수 있습니다.

## 부록 B. 재현 방법

- 코퍼스 원문 추출: `data/chroma_db/chroma.sqlite3`의 `embedding_metadata` 테이블에서 `book`/`chapter`/`verse`/`content` 직접 조회 (Chroma Python API 미사용)
- HuggingFace 임베딩: `langchain_huggingface.HuggingFaceEmbeddings(model_name="jhgan/ko-sroberta-multitask", device="cpu", normalize_embeddings=True)`
- OpenAI 임베딩: `langchain_openai.OpenAIEmbeddings(model="text-embedding-3-small")`
- 유사도: 정규화된 벡터의 내적(코사인 유사도), NumPy로 전수 계산 (풀 크기 16,500 × 질문 40개)
- 질문 생성: `gpt-4o-mini`, temperature=0.7
